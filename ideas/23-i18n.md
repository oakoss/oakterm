# Internationalization

Built for a global audience from day one. English is the default, but the architecture supports any language from the start.

## Two Separate Problems

### 1. Terminal Content (what programs output)

This is about rendering — the terminal must correctly display any text any program outputs, regardless of language:

- **CJK characters** — Chinese, Japanese, Korean wide characters render at double width. Handled by Unicode 16.0 grapheme cluster width (already in renderer spec).
- **Arabic / Hebrew** — right-to-left text. The VT protocol is inherently LTR, but BiDi rendering should handle display-only RTL correctly. Programs that output RTL text should render readably.
- **Devanagari, Thai, Khmer** — complex scripts with combining marks. Platform-native text shaping (HarfBuzz / Core Text) handles these correctly — this is why we defer to the platform.
- **Emoji** — grapheme clusters, ZWJ sequences, skin tone modifiers. Unicode 16.0 width tables + font fallback chain handles this.
- **CJK font fallback** — adding a CJK font to the fallback chain:

```lua
font = {
  family = "JetBrains Mono",
  fallbacks = {
    "Symbols Nerd Font",
    "Noto Sans CJK SC",    -- Simplified Chinese
    "Noto Color Emoji",
  },
}
```

This is core — not a plugin. Every terminal must render every script correctly.

### 2. Terminal UI (our chrome — sidebar, palette, settings, messages)

This is the part that can be extended:

- Sidebar section headers ("AGENTS", "SERVICES", "SHELLS")
- Command palette labels and descriptions
- Settings names and descriptions
- First-launch setup text
- Error messages and notifications
- `:debug` output labels
- `oakterm doctor` output
- CLI help text

## UI Localization Architecture

English ships in the core. Other languages are locale packs — lightweight plugins that provide translation strings.

```bash
oakterm plugin install locale-ja   # Japanese
oakterm plugin install locale-zh   # Chinese
oakterm plugin install locale-ko   # Korean
oakterm plugin install locale-es   # Spanish
oakterm plugin install locale-de   # German
oakterm plugin install locale-pt   # Portuguese
oakterm plugin install locale-fr   # French
```

### Locale pack structure

A locale pack is a WASM plugin that registers string translations:

```toml
# oakterm-plugin.toml
[plugin]
name = "locale-ja"
version = "1.0.0"
description = "Japanese localization for OakTerm"
type = "locale"
locale = "ja"

[capabilities]
# Locale packs need no special permissions
```

The plugin provides a translation map:

```json
{
  "sidebar.agents": "エージェント",
  "sidebar.services": "サービス",
  "sidebar.watchers": "ウォッチャー",
  "sidebar.shells": "シェル",
  "palette.search": "検索",
  "settings.font_family": "フォントファミリー",
  "settings.font_size": "フォントサイズ",
  "settings.theme": "テーマ",
  "settings.appearance": "外観",
  "debug.memory_overview": "メモリ概要"
}
```

### How it works

1. Core ships with English string keys for all UI text
2. If a locale plugin is installed and active, it overrides the English strings
3. Missing translations fall back to English — a partial translation is fine
4. System locale is detected automatically, locale plugin activated if installed
5. Override manually: `locale = "ja"` in config

### Plugin UI localization

Plugins that add UI (sidebar sections, palette commands) provide their own English strings. Locale packs can include translations for popular plugins:

```json
{
  "plugin.docker-manager.sidebar.containers": "コンテナ",
  "plugin.agent-manager.status.working": "作業中",
  "plugin.agent-manager.status.needs_input": "入力待ち"
}
```

## Configuration

```ini
locale = system   # auto-detect from OS (default)
locale = en       # force English
locale = ja       # force Japanese
```

In the settings palette, locale shows available options based on installed locale packs.

## What We Ship Day 1

- Core renders all Unicode correctly (CJK, RTL, emoji, complex scripts)
- All UI text uses string keys (ready for localization)
- English strings in the core
- Locale pack plugin API defined and documented
- Community can start contributing translations immediately

## What We Don't Do

- Machine translation — all locale packs are human-translated
- Locale-specific behavior changes (date formats, etc.) — the terminal doesn't display dates, programs do
- RTL UI layout — the terminal chrome stays LTR, only terminal content respects BiDi
