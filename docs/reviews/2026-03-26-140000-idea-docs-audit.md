# Idea Docs Audit

Full review of all 36 idea docs in `ideas/`, plus competitive research against Ghostty, Kitty, Alacritty, WezTerm, Warp, Rio, and Zellij. Covers factual accuracy, internal consistency, design gaps, and validation of key technical decisions.

## Scope

- **Internal audit**: Every idea doc reviewed for concreteness, contradictions, cross-reference accuracy, and claims needing validation.
- **Competitive research**: Current state of 7 competing terminals plus deep dives on wgpu, AccessKit, Wasmtime, and screen reader support in GPU terminals.

### Contradictions

Each of these needs an ADR to resolve.

#### 1. Kitty Graphics: Core or Plugin?

- [01-architecture.md](../../ideas/01-architecture.md) lists "Kitty graphics rendering" as a **plugin**
- [02-renderer.md](../../ideas/02-renderer.md) describes it as part of the **GPU pipeline (core)**, composited alongside text
- [03-multiplexer.md](../../ideas/03-multiplexer.md) says both image protocols "work through the multiplexer" (implies core)
- **Impact**: Determines whether the renderer needs image compositing primitives or delegates to a plugin

#### 2. Zero Telemetry vs Update Check

- [21-security.md](../../ideas/21-security.md) says "the terminal never communicates with any server"
- [24-updates.md](../../ideas/24-updates.md) describes an HTTP GET to a version manifest on every launch
- **Impact**: Privacy stance, trust, whether update check is opt-in or opt-out by default

#### 3. Performance Budget Oversubscribed

- [12-performance.md](../../ideas/12-performance.md) says "no feature may add >0.5ms to input latency"
- [17-accessibility.md](../../ideas/17-accessibility.md) budgets "<0.5ms/frame" for accessibility
- [14-debugging.md](../../ideas/14-debugging.md) shows plugin overhead of 1.31ms across 5 plugins, per-plugin budget of 0.5ms
- These are additive in reality. If a11y takes 0.5ms, plugins take 1.3ms, core rendering takes 4.9ms = 6.7ms total
- **Impact**: Need per-component budgets that add up to the 8ms target, not independent 0.5ms claims

#### 4. Context Engine: Sidecar vs WASM Plugin

- [05-context-engine.md](../../ideas/05-context-engine.md) describes a **Rust sidecar daemon** as a separate process
- [06-plugins.md](../../ideas/06-plugins.md) says **WASM is the plugin model**
- The doc calls it a "bundled plugin" but describes a separate process
- **Impact**: Sets precedent for whether plugins can be native sidecars or must be WASM-only

#### 5. Two Permission Systems

- [06-plugins.md](../../ideas/06-plugins.md): WASM capability model with kebab-case (`pane-create`, `fs-read`)
- [32-agent-control-api.md](../../ideas/32-agent-control-api.md): CLI permission model with snake_case (`pane_create`, `pane_read`)
- Both gate overlapping operations at different granularity (per-plugin vs per-pane)
- **Impact**: Need a unified permission model or explicit documentation of how the two systems interact

#### 6. Harpoon Persistence Model

- [27-harpoon.md](../../ideas/27-harpoon.md) says bookmarks "persist across sessions — part of the session serialization"
- Same doc's implementation section says it uses `storage` API (plugin-local persistent storage)
- These are two different persistence mechanisms
- **Impact**: Determines whether harpoon state lives in session files or plugin storage

#### 7. Scroll Buffer: Zero-Copy Claim vs Compressed Archives

- [12-performance.md](../../ideas/12-performance.md) says scrolling "doesn't allocate or copy — adjusts a viewport offset"
- [15-memory-management.md](../../ideas/15-memory-management.md) describes compressed disk archives with memory-mapped access
- Scrolling back into archived/compressed regions requires decompression — not zero-copy
- **Impact**: Need to clarify that zero-copy applies only to the hot ring buffer, not archived scrollback

#### 8. Agent State Detection: Two Mechanisms

- [07-agent-management.md](../../ideas/07-agent-management.md) says agents report status through "process output" with "provider-specific detection patterns"
- [32-agent-control-api.md](../../ideas/32-agent-control-api.md) lets agents explicitly set status via `phantom ctl self set-status`
- No precedence defined when they conflict
- **Impact**: Need to define which source of truth wins and how they compose

#### 9. 24-Hour Soak Test in CI

- [15-memory-management.md](../../ideas/15-memory-management.md) says "every PR runs" a list that includes the 24-hour soak test
- [25-testing.md](../../ideas/25-testing.md) correctly separates soak tests into "nightly extended runs"
- **Impact**: Memory doc needs updating to match testing doc

#### 10. Lua: Config Language or Extension Runtime?

- [01-architecture.md](../../ideas/01-architecture.md) puts "Lua config engine" inside the Extension Runtime alongside the WASM plugin host
- [09-config.md](../../ideas/09-config.md) treats Lua as declarative configuration
- But config examples include event handlers (`workspace.on_create = function(ws)`) which are imperative and long-lived
- **Impact**: Need to define the boundary — what can Lua do that WASM can't, and vice versa?

### Validated Decisions

Confirmed by competitive research (March 2026).

| Decision                         | Validation                                                                                                                                           |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Rust**                         | Alacritty, WezTerm, Rio all prove it works. Ecosystem (tokio, wasmtime, accesskit, mlua) is unmatched for this use case                              |
| **wgpu for rendering**           | Rio proves it works for terminals. Portability across Metal/Vulkan/DX12 from one codebase outweighs small abstraction overhead                       |
| **Platform-native text shaping** | Every terminal that rolls its own gets font complaints. Core Text / HarfBuzz is the right call                                                       |
| **Built-in multiplexer**         | Desktop trend is clearly toward built-in. WezTerm, Kitty, Ghostty, Zellij all offer it. tmux remains for servers only                                |
| **WASM plugins via Wasmtime**    | Zellij proves this works in production. Cold start <1ms. Near-native execution speed. Polyglot advantage over Kitty's Python-only kittens            |
| **Flat + Lua two-tier config**   | Ghostty's flat config praised for simplicity. WezTerm's Lua praised for power, criticized for complexity on basics. Two-tier captures both audiences |
| **Zero telemetry / no login**    | Warp's trust backlash (closed source, Segment telemetry, VC-funded) directly validates this stance                                                   |
| **AccessKit for accessibility**  | No GPU terminal does this. Proven in egui. Genuine competitive white space                                                                           |
| **Kitty graphics protocol**      | Becoming de facto standard. Adopted by Ghostty, WezTerm, Konsole, Warp, iTerm2                                                                       |
| **Ligatures from day one**       | Alacritty's decade-long refusal is now a meme. Every competitor ships them                                                                           |
| **MPL 2.0 license**              | Correct balance — core stays open, plugins can be any license, compatible with Apache 2.0 and GPL                                                    |
| **`xterm-256color` as TERM**     | Kitty's custom `xterm-kitty` TERM is a perennial SSH pain point. Using the standard TERM avoids this entirely                                        |

### Challenged Decisions

Research suggests these need revisiting or deeper analysis.

#### Accessibility in Phase 5

The roadmap puts a11y audit in Phase 5 (polish). Retrofitting accessibility into a GPU renderer is architecturally painful — the screen is a texture, not a DOM. Windows Terminal built UIA into the core from the start. AccessKit integration should be **Phase 0**, not Phase 5. The a11y tree must be built alongside the renderer.

#### Shell Integration in Phase 3

Scroll-to-prompt, per-command search, process notifications, and the context engine all depend on shell integration (OSC 133). Phase 3 is late for features users will expect in a daily-driver terminal that aspires to replace tmux (Phase 1). Consider moving basic shell integration to Phase 1.

#### <8ms Latency Target

Alacritty achieves 2-5ms with raw OpenGL. Ghostty uses direct Metal/OpenGL, not wgpu. wgpu adds an abstraction layer. Rio (wgpu) has had rendering issues. The target may need:

- Per-component budget breakdown (VT parse + shape + atlas + GPU submit + a11y)
- Benchmarking against Rio to validate wgpu overhead
- Tighter target (match Alacritty's 5ms, not just beat Ghostty's ~7ms)

#### <30MB Memory Target

- Wasmtime alone can use 10-15MB baseline
- wgpu device state uses memory for GPU context
- Glyph atlas defaults to 64MB max (per 15-memory-management.md)
- AccessKit tree adds overhead
- Zellij settled at 22MB _without_ a GPU renderer
- The 30MB target may require lazy WASM module loading and a nearly-empty initial glyph atlas, which could cause jank on first render

#### Predictive SSH Echo

Presented as a near-certain feature, but without Mosh's server-side component, client-only prediction is limited to the simplest cases (echoing typed characters). Tab completion, history expansion, and custom prompts can't be predicted. The feature may feel inconsistent enough to frustrate users. Needs deeper analysis or more conservative scoping.

#### 5ms Boot Time with Plugins

The startup path includes: WASM runtime init, Lua VM, plugin scanning, sidecar daemon (context engine), sidebar data model population, and shell integration setup. Needs a proof-of-concept benchmark before committing to this target.

#### libghostty as Ecosystem Threat

Ghostty extracted its core as a standalone C-ABI library (`libghostty`). Others could build terminals on it, potentially commoditizing the renderer layer. Phantom's differentiation (multiplexer, plugins, accessibility, config programmability) should be emphasized to stay ahead.

## Competitive Landscape Summary

As of March 2026:

| Terminal      | Stars | GPU            | Multiplexer        | Plugins           | Config         | A11y | Windows       |
| ------------- | ----- | -------------- | ------------------ | ----------------- | -------------- | ---- | ------------- |
| **Ghostty**   | 45K+  | Metal/OpenGL   | Basic tabs/splits  | None              | Flat key=value | None | No            |
| **Kitty**     | —     | OpenGL         | Tabs/splits        | Python (kittens)  | Custom conf    | None | No            |
| **Alacritty** | —     | OpenGL         | None               | None              | TOML           | None | Yes           |
| **WezTerm**   | —     | OpenGL         | Full (SSH domains) | None (Lua config) | Lua            | None | Yes (stalled) |
| **Warp**      | —     | Metal          | Tabs               | None              | GUI            | None | Yes           |
| **Rio**       | —     | wgpu           | Basic              | None              | TOML           | None | Yes           |
| **Zellij**    | —     | N/A (mux only) | Full               | WASM (Rust-only)  | KDL            | None | No            |

**Key openings for Phantom:**

- No GPU terminal has screen reader support (AccessKit opportunity)
- No terminal combines wgpu + built-in multiplexer + WASM plugins
- WezTerm's maintenance vacuum leaves its ambitious feature set unserved
- Ghostty has no Windows support, no plugins, no session persistence
- Zellij's WASM plugins are Rust-only; Phantom's polyglot approach is a differentiator
